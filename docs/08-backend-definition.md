# 08. バックエンド定義

## 目的

本書は、既存の要件・画面設計・状態遷移を前提に、v1 でバックエンドが担う責務を定義するための文書である。  
ここでいうバックエンドとは、外部サーバーではなく、`VS Code Extension Host` 上で動作するアプリケーション層・文脈収集層・モデル呼び出し層・ナレッジ層を指す。

## 前提

- v1 では専用のリモートバックエンドは持たない
- AI 呼び出しは `GitHub Copilot + VS Code Language Model API` を利用する
- 保存先はローカル環境であり、個人用データのみを扱う
- UI は `WebviewView` を利用し、バックエンドは UI に対して状態と実行結果を提供する

## バックエンドの責務

| 領域 | 責務 | 対応する仕様 |
|---|---|---|
| 接続管理 | Copilot 利用可否判定、接続開始、同意フロー通過、接続状態更新 | 02.2, 04, 06 |
| 状態管理 | 画面表示用の状態を一元管理し、UI に返す | 05, 06, 07 |
| 文脈収集 | 送信候補となる作業文脈を収集し、除外条件を適用する | 02.1, 02.4, 03.1, 03.2 |
| 送信計画 | 今回送る文脈、送信理由、送信量、根拠情報を組み立てる | 02.6, 03.1, 03.3, 05.4 |
| AI 実行 | 手動相談・常時助言・深掘り質問をモデルへ送信し、応答を返す | 02.1, 02.2, 02.3, 05.2, 05.3 |
| 発話制御 | 間隔、アイドル時間、重複抑制、一時停止を制御する | 02.7, 05.2, 05.6, 06 |
| 会話管理 | 会話履歴、最新アドバイス、詳細表示用データを保持する | 05.2, 05.3 |
| ナレッジ管理 | 個人ナレッジの保存、検索、編集、無効化、削除、出力、リセットを行う | 02.5, 03.4, 03.6, 05.5 |
| 設定管理 | モード、送信対象、除外設定、頻度制御設定などを保存・適用する | 03.3, 05.6 |
| 失敗時制御 | 制限状態への遷移、再試行可否判定、回復後の復帰処理を行う | 03 失敗時挙動, 06 |

## バックエンドの実装対象

### 8.1 接続管理

バックエンドは以下を提供する。

- 現在状態の取得
- `Copilotに接続` 実行
- `Trusted Workspace`、モデル取得可否、同意結果の判定
- 初回接続 probe の実行
- 接続失敗理由の分類

必要な内部処理は以下とする。

- `disconnected / connecting / consent_pending / connected / restricted / unavailable` の状態管理
- 接続開始中の多重実行防止
- 同意キャンセル時の未接続復帰
- quota 超過、一時制限、offline、untrusted の分類

### 8.2 画面状態の集約

バックエンドは UI ごとの生データではなく、画面表示可能な ViewModel を返す。

- S01 用: 接続状態、送信範囲要約、送信除外要約、利用条件案内、エラー文言
- S02 用: モード、会話履歴、入力可否、参照文脈サマリ、最新アドバイス、クールダウン状態
- S03 用: 助言本文、根拠一覧、変更文脈、推測表示、深掘り可能状態
- S04 用: 送信対象カテゴリ、対象ファイル一覧、推定送信量、除外理由一覧
- S05 用: ナレッジ一覧、詳細、検索結果、状態フィルタ結果
- S06 用: 現在設定値、有効値、保存可否
- S07 用: エラー種別、説明、推奨アクション、再試行可否

### 8.3 文脈収集

バックエンドは、AI 送信前に以下の文脈を収集する。

- アクティブファイル断片
- 選択範囲
- diagnostics
- 最近の編集範囲
- 関連シンボル
- 必要に応じたファイル構造、定義と参照、最近の変更内容

文脈収集では以下を満たす。

- ユーザー設定に応じて送信対象の有効 / 無効を切り替える
- 除外 glob、生成物、大容量ファイル、秘密情報候補を除外する
- UI に表示するため、収集結果に「採用理由」と「除外理由」を持たせる
- トークン量または文字量の上限に収まるように優先順位を付ける

### 8.4 送信計画と根拠生成

バックエンドは、モデルに送る前に `RequestPlan` を構築する。

`RequestPlan` には最低限以下を含める。

- リクエスト種別
- 利用モード
- 採用された文脈の一覧
- 除外された文脈の一覧
- 推定送信量
- 根拠表示用の参照元情報
- 重複抑制判定用の特徴量

リクエスト種別は以下を想定する。

- 初回接続 probe
- 手動相談
- 常時助言
- 深掘り質問
- ナレッジ候補化のための要約

### 8.5 AI 実行

バックエンドは、用途に応じてプロンプトと送信コンテキストを組み立て、モデル呼び出しを実行する。

- 手動相談: ユーザーの明示操作で実行
- 常時助言: トリガー条件を満たしたときのみ実行
- 深掘り質問: 直前の助言と会話履歴を引き継いで実行

AI 実行で必要な要件は以下とする。

- 学習支援特化のプロンプトを固定方針として適用する
- 参照した文脈を応答結果に紐付ける
- 応答をストリーミング可能な形で UI に渡せるようにする
- 応答失敗時に状態遷移とエラー情報を返す

### 8.6 発話制御

常時モードのために、バックエンドはスケジューラを持つ。

- 最小リクエスト間隔
- アイドル時間
- 直近応答との類似度判定
- 制限中の自動停止
- 一時停止 / 再開

トリガー候補は以下とする。

- テキスト編集
- アクティブエディタ変更
- diagnostics 変化
- アイドル時間経過

ただし、トリガー発火時に毎回送信するのではなく、`発火判定 -> 送信計画生成 -> クールダウン判定 -> 実行` の順で判定する。

### 8.7 会話とアドバイス履歴

バックエンドは以下を管理する。

- 会話履歴
- 最新アドバイスカード
- アドバイス詳細表示用レコード
- 直前応答への深掘り元情報

履歴 1 件あたりに保持するデータ例は以下とする。

- 種別: 手動相談 / 常時助言 / 深掘り
- 発生日時
- ユーザー入力
- AI 応答本文
- 根拠一覧
- 参照文脈要約
- 推測を含むか
- ナレッジ候補化状態

### 8.8 ナレッジ管理

バックエンドは SQLite を用いて個人ナレッジを管理する。

必要な機能は以下とする。

- ナレッジ作成
- ナレッジ検索
- ナレッジ編集
- ナレッジ無効化 / 再有効化
- ナレッジ削除
- JSON / Markdown エクスポート
- 全件リセット

最低限必要な保存項目は以下とする。

- `id`
- `title`
- `summary`
- `body`
- `status`
- `tags`
- `sourceAdviceId`
- `createdAt`
- `updatedAt`

加えて、AI 送信時には以下の処理が必要になる。

- 現在文脈に関連するナレッジ候補の検索
- 有効なナレッジのみを再利用対象とする
- 送信量上限に応じた取り込み件数制御

### 8.9 設定管理

バックエンドは設定の読み書きと有効化を行う。

管理対象は以下とする。

- 初期モード
- 常時モード有効可否
- リクエスト間隔
- アイドル時間
- 重複抑制
- 送信対象設定
- 除外 glob
- 大容量閾値
- ナレッジ再利用 ON / OFF
- 送信前確認 ON / OFF

設定は以下の 2 層で扱う。

- 永続設定: `workspaceState` または `globalState`、必要に応じて `vscode.workspace.getConfiguration`
- 実行時設定: 現在反映済みのスケジューラ・収集条件・UI 表示用設定

### 8.10 失敗時制御

バックエンドは、失敗時のエラーを UI 向けの状態へ正規化する。

扱うべき代表状態は以下とする。

- 同意未取得
- Copilot 利用不可
- offline
- quota 超過
- 一時エラー
- untrusted workspace

必要な処理は以下とする。

- S07 表示用のメッセージ生成
- 再試行可否判定
- 常時モードの自動停止
- 制限復旧後の必要時モード復帰

## バックエンド内部モジュール案

v1 では以下のモジュール分割を推奨する。

- `ConnectionService`: Copilot 接続と利用状態管理
- `SessionStore`: 画面状態、会話状態、最新アドバイスの保持
- `ContextCollector`: 文脈収集
- `ContextFilter`: 除外条件適用、送信上限制御
- `RequestPlanner`: 送信計画と根拠情報の生成
- `AdviceScheduler`: 常時モードのトリガー制御
- `AdviceService`: プロンプト生成、モデル呼び出し、応答整形
- `KnowledgeStore`: SQLite CRUD
- `KnowledgeRetriever`: 文脈関連ナレッジの検索
- `SettingsService`: 設定の永続化と反映
- `ErrorMapper`: モデルエラーや環境エラーの正規化

## 優先実装順

### Phase 1

- 接続管理
- 手動相談フロー
- 基本的な文脈収集
- 画面状態の ViewModel 化

詳細設計は `docs/09-phase1-implementation-design.md` を参照する。

### Phase 2

- 送信計画と送信範囲確認
- 会話履歴
- アドバイス詳細と根拠表示
- 設定管理

### Phase 3

- 常時モード用スケジューラ
- クールダウン表示
- 重複抑制
- 一時停止 / 再開

### Phase 4

- SQLite ナレッジ保存
- ナレッジ検索・編集・削除
- ナレッジ再利用
- エクスポート / リセット

## 現在コードとの差分

現状コードでは以下が未充足である。

- 会話履歴を保持していない
- 文脈収集が `active file / selection / diagnostics` に限定されている
- 送信計画、送信量計算、除外理由生成がない
- 常時モードのトリガー制御がない
- ナレッジ保存が未実装
- 設定値の永続化と反映が限定的である
- エラー状態が S07 向けの画面状態へ十分に正規化されていない

## バックエンドの非責務

v1 のバックエンドは以下を行わない。

- 外部サーバーとの同期
- チーム共有ナレッジ
- Copilot 認証情報の保存
- AI によるコード変更やコマンド実行
- Web 版 VS Code 対応
