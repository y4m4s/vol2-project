# 11. ロウモード / ハイモード仕様

- 文書種別: 仕様書ドラフト
- 対象: VS Code Extension v1
- 作成日: 2026-05-18
- ステータス: 実装反映版

## 概要

ロウモード / ハイモードは、NaviCom の助言の深さを切り替えるための機能である。  
目的は、ユーザーの学習支援というプロダクト方針を保ったまま、状況に応じて「短いヒント」と「やや構造化された整理」を使い分けられるようにすることである。

NaviCom はコードの自動修正や実装代行を行わない。  
ハイモードでも、完成コードや最終回答を直接提示するのではなく、確認順、判断材料、注意点、次の検証観点を厚めに返す。

## モード定義

| モード | 用途 | 応答方針 |
|---|---|---|
| ロウ | 通常時、学生ユーザー、Copilot 利用上限を意識した利用 | 短いヒント、確認ポイント、見る場所を中心に返す |
| ハイ | 詰まりが深いとき、一区切り後の整理、設計判断が必要なとき | 確認順、トレードオフ、境界条件、検証観点を構造化して返す |

常時モードでは、発話頻度と送信量を抑えるため、助言の深さは実質的にロウ相当として扱う。  
手動相談では、ユーザーが UI からロウ / ハイを切り替えられる。

## 現状の実装

### 型と状態

`AssistanceDepth` として以下の 2 値を持つ。

```ts
export type AssistanceDepth = "low" | "high";
```

主な保持箇所は以下である。

- `NavigatorSettings.defaultAssistanceDepth`
- `NavigatorSessionState.assistanceDepth`
- `NavigatorViewModel.assistanceDepth`
- `RequestPlanSnapshot.assistanceDepth`
- `GuidanceCard.assistanceDepth`
- `ConversationEntry.assistanceDepth`

これにより、現在の UI 状態、リクエスト計画、会話履歴、保存済み相談のいずれでも、どの深さで生成された助言かを扱える。

### 設定

設定画面では、既定の深さを `ロウ` / `ハイ` から選択できる。  
保存先は VS Code のローカル `Memento` であり、既定値は `low` である。

設定初期化時は以下に戻る。

- 初期モード: `manual`
- 既定の深さ: `low`
- アイドル時間: `10000ms`
- 複数ファイル・ディレクトリ構造: `false`

### 入力 UI

チャット入力欄の右側に深さ切り替えボタンを表示する。

- ロウ時: `lightbulb` アイコンと `ロウ`
- ハイ時: `travel_explore` アイコンと `ハイ`

リクエスト実行中は切り替え不可とする。  
切り替えは即座に設定へ保存され、以降の手動相談に反映される。

### プロンプト

`AdviceService` は、生成時に以下の応答ルールをプロンプトへ含める。

ロウモード:

- 短いヒントと確認ポイントを中心にする
- 長い説明を避ける
- 最終回答へ飛ばない

ハイモード:

- 確認順、トレードオフ、境界条件を含める
- ロウより深く整理する
- ただし実装代行や完成コード提示は避ける

### 送信コンテキスト

深さは応答方針だけでなく、送信する文脈の量にも反映される(`RequestPlanner`)。

| 深さ | 送信コンテキスト |
|---|---|
| ロウ(常時モード含む) | アクティブファイル断片(最大2000字)、選択範囲、Diagnostics(最大5件)、最近の編集(最大5件)、関連シンボル(最大8件)。関連ファイル・ディレクトリ構造は送信しない |
| ハイ | 上記の制限なし。設定で有効な場合は関連ファイル・ディレクトリ構造も送信する |

常時モードは深さがロウ相当で固定のため、常にロウの縮約が適用される。  
選択範囲・追加コンテキスト・プロジェクト概要(`/next`)は深さによる縮約の対象外とする。

### 会話履歴

会話画面では、AI 応答側に `ロウ` / `ハイ` のメタ情報を表示する。  
会話履歴を SQLite に保存する際も `assistance_depth` として保存する。

## スラッシュコマンドとの関係

スラッシュコマンドの応答も、現在の深さに影響を受ける。

例:

- `/hint` + ロウ: 2-3 個の短いヒント
- `/hint` + ハイ: 原因候補と確認順を 3-5 個で整理
- `/next` + ロウ: 次に確認するとよいことを 3 個以内で短く提示
- `/next` + ハイ: 完了確認、検証、次の実装候補、後回しでよいことを分けて整理

`/next` では、深さがプロジェクト文脈の厚みにも影響する。

| 入力 | 深さ | プロジェクト文脈 |
|---|---|---|
| `/next` | ロウ | `project-lite` |
| `/next` | ハイ | `project` |
| `/next deep` | ロウ / ハイ | `deep` |

## 送信量の方針

ロウ / ハイは単なる文章量の切り替えではなく、送信量と推論量のバランスにも関係する。

基本方針:

- 通常はロウを既定にする
- 学生ユーザーや Copilot Student を想定し、軽量モデル・少ない送信量でも成立する設計にする
- ハイは明示的に深く相談したいときに使う
- 常時モードはロウ相当で固定し、過剰なトークン消費を避ける
- プロジェクト全体を広く見る処理は、原則として `/next` など目的が明確なコマンドに限定する

## 現状の制約

- ロウ / ハイはモデル選択を直接変更しない
- Copilot 側の利用可能モデルや上限は VS Code Language Model API と Copilot の状態に依存する
- ハイモードでも、プロジェクト全体を常に深く読むわけではない
- 常時モードでは、深い説明よりも短い見落とし指摘を優先する

## 今後の展開

### 11.1 モード別の送信計画表示

現状のリクエスト計画には推定サイズとカテゴリがある。  
今後は、ロウ / ハイによって何が増減するかを UI 上でより明示する。

例:

- ロウ: アクティブファイル、選択範囲、Diagnostics、最近の編集
- ハイ: 上記に加えて関連ファイル、ディレクトリ構造、プロジェクト概要

### 11.2 コマンド単位の推奨深さ

コマンドごとに推奨深さを持たせる。

例:

- `/hint`: ロウ推奨
- `/flow`: ハイ推奨
- `/next`: ハイまたは `/next deep` 推奨
- `/test`: ハイ推奨

ただし、自動で切り替えるのではなく、まずは UI 上の補助表示に留めるのがよい。

### 11.3 予算モード

Copilot の仕様変更や利用上限を考慮し、将来的には以下のような予算モードを検討する。

| 予算モード | 方針 |
|---|---|
| 節約 | ロウ固定、関連ファイル少なめ、常時モード控えめ |
| 標準 | 現在の既定動作 |
| 詳細 | ハイを使いやすくし、必要時にプロジェクト文脈を厚くする |

ロウ / ハイは応答の深さ、予算モードは送信量と頻度の制御として分けると整理しやすい。

### 11.4 回答品質のフィードバック

回答ごとに簡易フィードバックを付け、ロウ / ハイのどちらで有用だったかを学習できるようにする。

例:

- ちょうどよい
- 短すぎる
- 詳しすぎる
- 文脈が足りない

この情報はローカルに保存し、将来的なプロンプト調整や既定深さの提案に利用できる。

## 判断基準

ロウ / ハイの設計では、以下を優先する。

- 学習支援であること
- ユーザーの作業を奪わないこと
- 送信量を必要以上に増やさないこと
- 深い相談が必要なときだけ、明示的に厚い文脈を使うこと
- Copilot のプラン差や上限変更に耐えやすいこと
