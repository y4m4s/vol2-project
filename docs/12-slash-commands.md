# 12. スラッシュコマンド仕様

- 文書種別: 仕様書ドラフト
- 対象: VS Code Extension v1
- 作成日: 2026-05-18
- ステータス: 実装反映版

## 概要

スラッシュコマンドは、ユーザーが「どの観点で相談したいか」を短く指定するための入力補助機能である。  
自然文で毎回説明しなくても、詰まりのヒント、次の一手、処理の流れ、リスク、テスト観点をすばやく呼び出せるようにする。

NaviCom の方針上、スラッシュコマンドは実行命令ではなく、助言の観点を切り替えるためのものとして扱う。  
コマンド実行後も、AI はコード修正やコマンド実行を代行せず、ユーザーが自分で次の行動を選べるように支援する。

## 現状のコマンド一覧

| コマンド | 用途 | 主な返答 |
|---|---|---|
| `/hint` | 詰まりをほどく | 見る場所、確認する問い、短いヒント |
| `/next` | 作業の一区切り後に次の行動を整理する | 完了確認、検証、次の実装候補、後回しでよいこと |
| `/next deep` | `/next` を広めのプロジェクト文脈で実行する | プロジェクト概要を踏まえた優先順位と根拠 |
| `/flow` | 処理やデータの流れを整理する | 入力、変換、出力、副作用の流れ |
| `/risk` | 壊れやすい箇所や副作用を確認する | 境界条件、影響範囲、見落としやすい点 |
| `/test` | 確認観点を整理する | 正常系、境界値、失敗系、回帰確認 |

## UI 仕様

### 入力欄でのサジェスト

チャット入力欄で `/` を入力すると、入力欄の上にスラッシュコマンド候補を表示する。  
候補は `SlashCommandSuggest` コンポーネントで実装する。

表示内容:

- コマンド文字列
- 表示名
- 説明
- Material Symbols アイコン

操作:

- クリック: 当該コマンドを実行
- `ArrowDown`: 次の候補へ移動
- `ArrowUp`: 前の候補へ移動
- `Enter`: 選択中の候補を実行
- `Escape`: サジェストを閉じる

入力が `/next d` のように途中まで入っている場合も候補検索の対象にする。  
複数行入力ではサジェストを表示しない。

### スラッシュコマンドボタン

初見ユーザーが機能に気づけるよう、入力欄下部のコンテキストボタンの右に `/` ボタンを表示する。

仕様:

- ボタンは場所を取らないアイコン風の表示にする
- 押下するとスラッシュコマンド候補を開く
- もう一度押下すると閉じる
- リクエスト実行中は無効化する

## 入力解析

Controller 側では、送信テキストを以下のように解析する。

```text
/command optional arguments
```

既知のコマンドに一致した場合:

- `slashCommand` を設定する
- 残りの文字列を `userPrompt` として扱う
- 会話履歴にはコマンドに応じた自然文ラベルを表示する

未知のコマンドの場合:

- 通常の自然文相談として扱う
- エラーにはしない

`/next` のみ、追加引数として `deep`, `wide`, `full` を受け付ける。  
これらはすべて `slashCommandScope = "deep"` として扱う。

例:

| 入力 | 解釈 |
|---|---|
| `/next` | 標準の次の一手 |
| `/next deep` | 広めのプロジェクト文脈で次の一手 |
| `/next deep UI周りを中心に` | deep scope + 追加相談文 |
| `/foo` | 通常の相談文 |

## プロンプト仕様

`AdviceService` は、プロンプトに以下を含める。

- 現在の深さ
- スラッシュコマンド名
- `/next deep` の場合は deep scope
- コマンド別の応答指示

コマンド別の基本方針:

### `/hint`

- 答えを出し切らない
- 見る場所と問いを中心にする
- ロウでは 2-3 個、ハイでは確認順をやや厚めに出す

### `/next`

- 作業が一区切りついた前提で整理する
- ロウでは短い次の確認候補を返す
- ハイでは完了確認、検証、次の実装候補、後回し事項を分ける
- `/next deep` ではプロジェクト概要を根拠として使う

### `/flow`

- 処理やデータの流れを整理する
- 流れの整理には関連ファイル等の厚い文脈が必要なため、深さ設定に関わらず常にハイとして実行する
- Mermaid の `flowchart TD` を 1 つ含める
- 推測しすぎず、送られた文脈から分かる範囲に留める

### `/risk`

- 壊れやすい境界、副作用、影響範囲を整理する
- 断定しすぎず、確認ポイントとして提示する

### `/test`

- テストコードは書かない
- 何を確かめるべきかに絞る
- ハイでは正常系、境界値、失敗系、回帰確認に分ける

## `/next` の文脈収集

`/next` は、現在のファイルだけでは精度が出にくい。  
そのため、通常の相談よりもプロジェクト全体を薄く見る専用文脈を追加する。

### スコープ

| スコープ | 発動条件 | 方針 |
|---|---|---|
| `project-lite` | `/next` + ロウ | 送信量を抑えて最低限の全体像を見る |
| `project` | `/next` + ハイ | 少し広めに全体像と関連ファイルを見る |
| `deep` | `/next deep` | 明示的に広い整理を求めた場合のみ使う |

### 収集対象

`collectNextActionContext()` は、以下を収集して `projectSummary` にまとめる。

- ディレクトリ構造
- 関連ファイル断片
- 開いているファイル一覧
- ワークスペース全体の Diagnostics 要約
- 最近の編集要約
- TODO / FIXME / XXX
- `package.json`, `tsconfig.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`
- `README.md` と `docs/**/*.md`

いずれも全文送信ではなく、件数と文字数を制限した要約として扱う。

### 除外設定

`/next` のプロジェクト文脈でも、保護済み除外 glob とユーザー指定除外 glob を適用する。  
依存物、生成物、DB、アーカイブ、認証情報を含みやすいファイルは送信対象外にする。

## 保存仕様

会話履歴には以下を保存する。

- `assistance_depth`
- `slash_command`
- `slash_command_scope`
- `request_plan_json`
- `guidance_context_json`

これにより、過去の回答がどのコマンド、どの深さ、どの文脈スコープで生成されたかを後から参照できる。

## 現状の制約

- コマンド候補のカスタマイズはできない
- `/next deep` 以外のサブコマンドは未実装
- コマンド別の送信範囲をユーザーが細かく調整する UI はない
- `/next` のプロジェクト理解はファイル構造と要約ベースであり、完全な依存グラフ解析ではない

`/flow` の Mermaid コードブロックは、会話画面で SVG として描画する(mermaid ライブラリをバンドル)。  
描画に失敗した場合はコードブロック表示にフォールバックする。

## 今後の展開

### 12.1 コマンドパレット化

スラッシュコマンド候補を、単なる一覧ではなく目的別に整理する。

例:

- 詰まりをほどく: `/hint`, `/risk`
- 作業を進める: `/next`, `/test`
- 理解する: `/flow`

### 12.2 コマンド別の文脈プリセット（実装済み）

コマンドごとに送信カテゴリを最適化する。スキルレジストリ（`SkillDefinition.contextPreset`）で宣言し、`RequestPlanner` が送信前に許可リスト外のカテゴリを落とす。関連性の向上と送信トークン削減を両立する。詳細は `docs/14`。

現在のプリセット:

| コマンド | 送る文脈カテゴリ |
|---|---|
| `/hint` | アクティブファイル、選択範囲、Diagnostics、最近の編集 |
| `/flow` | アクティブファイル、選択範囲、関連シンボル、関連ファイル、ディレクトリ構造 |
| `/risk` | アクティブファイル、選択範囲、Diagnostics、最近の編集、関連ファイル |
| `/test` | アクティブファイル、選択範囲、Diagnostics、最近の編集 |
| `/next` | アクティブファイル、選択範囲、Diagnostics、プロジェクト概要 |

`additionalContext`（ユーザー入力）は許可リストに関わらず常に送る。除外されたカテゴリはリクエストプラン上で理由表示される。

### 12.3 `/next` の精度向上

今後は以下を追加すると、`/next` の精度が上がる。

- Git の変更差分
- package scripts から推定した実行可能な検証コマンド
- テストファイル候補の抽出
- 最近開いた会話やナレッジの参照
- 直近回答に対する未完了アクションの追跡

ただし、送信量が増えやすいため、既定では軽量にし、明示的な deep 指定時のみ広げる。

### 12.4 コマンド履歴と再実行

よく使うコマンドを履歴から再実行できるようにする。

例:

- 直近の `/next`
- 直近の `/test`
- 同じ追加コンテキスト付きの `/hint`

### 12.5 コマンド結果の構造化

コマンドごとに、回答フォーマットをさらに安定させる。

例:

- `/next`: `完了確認`, `次に見る候補`, `後回しでよいこと`
- `/risk`: `高`, `中`, `低`
- `/test`: `正常系`, `境界値`, `失敗系`, `回帰確認`

構造化しすぎると会話らしさが落ちるため、まずはプロンプト上の緩い制約として扱う。

### 12.6 ユーザー定義コマンド

将来的には、ユーザーがローカルに独自コマンドを追加できるようにする。

例:

```yaml
/review-lite:
  title: 軽いレビュー
  prompt: 変更箇所の壊れやすい境界だけ短く指摘してください
```

v1 では対象外とし、まずは組み込みコマンドの品質を優先する。

## 判断基準

スラッシュコマンドの設計では、以下を優先する。

- 初見でも存在に気づけること
- 入力操作を邪魔しないこと
- コマンド数を増やしすぎないこと
- 学習支援の範囲を超えないこと
- `/next` など広い文脈が必要な処理は、送信量と精度のバランスを明示的に取ること
